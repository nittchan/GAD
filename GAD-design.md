## Problem Statement

Parametric insurance pays when a trigger fires — a weather threshold, a seismic reading, a rainfall measurement. The actuarial work behind that trigger (the math that decides when payouts happen and how well the trigger correlates with actual losses) is a black box. No one outside the insurer's actuarial team can audit it. GAD makes that math open-source, auditable, and standardized.

## What Makes This Cool

The "holy shit" moment: you pull up a live dashboard, pick any parametric trigger, and see its entire actuarial profile — Spearman basis risk scores, historical back-tests against open data, Lloyd's alignment rating — all transparent, all verifiable. The insurance industry's most opaque math, laid bare in a URL you can share.

The 10x vision compounds this: a global trigger registry where every parametric trigger ever deployed is cataloged with its basis risk score, like a CVE database for insurance. The dashboard is the window into the registry.

## Constraints

- Must use only open data sources (ERA5, CHIRPS, NOAA, etc.) — no proprietary weather or loss data dependencies
- Must align with Lloyd's standards for parametric product approval
- Spearman rank correlation is the core basis risk methodology
- Code is early-stage (methodology is clear, implementation is greenfield)
- Target audience: developers building parametric insurance products
- Builder project — optimizing for fun, learning, and "whoa" factor

## Premises

1. **Transparency over novelty.** The Spearman methodology already exists in proprietary tools (Willis Re, Aon, in-house models). Making it open and auditable is the breakthrough — not better math.
2. **Developers first.** Devs building parametric products need to trust the trigger math enough to build on it. A live dashboard proves that trust.
3. **Open data sufficiency.** Public climate/weather data (ERA5, CHIRPS, NOAA) is sufficient to compute meaningful basis risk scores without proprietary data.
4. **Lloyd's alignment is practical.** GAD's methodology maps to Lloyd's standards for parametric product approval — this helps get products to market, not just academic validation.

## Approaches Considered

### Approach A: Library-First
Build GAD as a Python library (pip install gad) with a CLI. Dashboard comes later.
- **Pros:** Clean API design, easy to test, devs can integrate via pip
- **Cons:** No visual demo, no "whoa" moment until later

### Approach B: Dashboard-First
Build the live audit dashboard as the primary artifact. Library emerges from backend refactoring.
- **Pros:** Immediate visual impact, shareable URL, fast feedback loop
- **Cons:** Risk of coupling UI to computation, more infrastructure

### Approach C: Registry-Anchored
Start with the global trigger registry as the core data model. Dashboard is a view into the registry.
- **Pros:** Aligns with 10x vision, forces standardization early, triggers are permanent assets
- **Cons:** More upfront design, schema definition scope creep

## Recommended Approach

**B+C: Registry-backed Dashboard**

Combine the dashboard's immediate visual impact with the registry's data model discipline. The architecture:

1. **Registry layer** — A database (SQLite for dev, Postgres for prod) with a standardized trigger schema: trigger definition, peril type, geography, data sources, parameters, Spearman basis risk score, confidence intervals, Lloyd's alignment rating, historical back-test results.

2. **Computation engine** — Python core that takes a trigger definition + open data source and produces: Spearman rank correlation coefficients, basis risk scores, confidence intervals, historical back-test against actual loss events, Lloyd's alignment assessment.

3. **Live dashboard** — Web app (Streamlit for v1, or Next.js for polish) that lets users:
   - Browse the trigger registry
   - Select any trigger and see its full basis risk profile
   - Visualize historical performance (trigger fires vs. actual losses)
   - Compare triggers side-by-side
   - Submit new triggers for analysis

4. **Open data pipeline** — Automated ingestion from ERA5, CHIRPS, NOAA, and other public sources. Keeps the registry's back-tests current.

**Why B+C:** The dashboard delivers the "whoa" (share a URL, devs see the math). The registry ensures every trigger analyzed becomes a permanent, auditable asset — building toward the CVE-for-insurance vision. And because the registry schema forces standardization early, it naturally aligns with Lloyd's requirements.

**Build order:**
1. Define the trigger schema (the registry's data model)
2. Build the Spearman basis risk computation engine (Python)
3. Wire up one open data source (start with ERA5 or CHIRPS)
4. Build the dashboard on top (Streamlit for speed)
5. Populate with example triggers (drought, flood, earthquake)
6. Ship a shareable URL

## Engineering Decisions (from /plan-eng-review)

- **Phased build:** Phase 1 = Engine + Dashboard with flat files. Registry + pipeline deferred to Phase 2.
- **Data flow:** Separate trigger YAML (pure insurance logic) from data manifest (maps triggers to data sources).
- **Lloyd's scoring:** Explicit checklist with boolean pass/fail per criterion. Score = fraction passing. Unknown criteria raise errors.
- **Spearman correlation:** Both spatial basis risk (always computable) and loss-proxy correlation (when NDVI/EM-DAT available).
- **Input validation:** yaml.safe_load + Pydantic schema validation. Bounded date ranges and spatial extent.
- **Models:** Pydantic everywhere (TriggerDef, DataManifest, BasisRiskReport, LloydsResult).
- **Engine API:** Functional — compute_basis_risk() and lloyds_check(). No classes.
- **Spatial scope:** Point-only (single lat/lon) in Phase 1. Regional triggers deferred.

## Dashboard Design (from /plan-design-review)

### Layout
```
┌─────────────────────────────────────────────────────────┐
│  HEADER: GAD logo + tagline                    [Upload] │
├─────────────┬───────────────────────────────────────────┤
│  SIDEBAR    │           MAIN PANEL                      │
│  Trigger    │  1st: Score Card (ρ, CI, Lloyd's score)   │
│  List       │  2nd: Back-test Timeline + Scatter Plot   │
│  + Upload   │  3rd: Confusion Matrix + Lloyd's Checklist│
│             │  4th: Expandable Methodology Details      │
└─────────────┴───────────────────────────────────────────┘
```

### Score Card
- Three headline numbers: Spearman ρ, confidence interval, Lloyd's pass rate (e.g., 8/10)
- P-value in expandable details section
- Monospace font for numbers. Colors: green (ρ > 0.7), amber (0.4–0.7), red (< 0.4)

### Visualizations
- **Dual-strip timeline:** Top strip = trigger fires (binary per period). Bottom strip = actual losses. Mismatches highlighted in red. Shows "when did it go wrong?"
- **Scatter plot with quadrants:** X = trigger variable, Y = loss proxy. Dots = time periods. Quadrants = TP/FP/FN/TN. Shows "how correlated?"

### Lloyd's Checklist
- Each criterion: check name | pass/fail badge | one-line explanation
- Failed checks get red left border. Expandable detail on click.
- Unknown/unimplemented criteria raise explicit errors (never silently skip)

### Trigger Upload
- Inline YAML code editor with syntax highlighting (st_ace or st_code_editor)
- Validation errors as inline annotations, not alert boxes
- Playground feel: tweak parameters and re-compute instantly

### Comparison View
- Max 2 triggers. Multi-select in sidebar → main panel splits into two columns
- Inline split-panel (no page transition). Delta values highlighted between profiles

### First-Run Experience
- Sample Kenya drought trigger auto-loaded and pre-computed on first visit
- "Whoa" moment in under 10 seconds — no setup required

### Interaction States
| Feature | Loading | Empty | Error | Success | Partial |
|---------|---------|-------|-------|---------|---------|
| Trigger list | Skeleton cards | "No triggers yet. Upload your first →" | "Failed to load. Check data/ directory." | List renders | N/A |
| Basis risk | Spinner + "Computing..." | N/A | "Computation failed: [reason]" | Full 4-tier profile | Spatial only (no proxy) |
| YAML upload | "Validating..." | N/A | Inline field errors | "Loaded" + auto-compute | Warning for missing optional fields |
| Back-test | Skeleton chart | "No historical data" | "Insufficient data (need ≥10)" | Timeline renders | Zero fires → warning |
| Lloyd's checklist | Shimmer per item | N/A | "Cannot evaluate [criterion]" | Green/red per check | Inconclusive → yellow "?" |
| Comparison | Both columns spinner | "Select second trigger" | Error in failed column only | Split panel | N/A |

### Visual Language (minimum viable)
- **Font:** System sans-serif for UI, monospace (JetBrains Mono / Fira Code) for data
- **Colors:** Dark mode default. Accent: teal/cyan. Semantic: green/amber/red
- **Spacing:** 8px grid
- **Charts:** Plotly with dark theme. No 3D effects.
- **Anti-slop:** Bloomberg terminal aesthetic, not generic SaaS cards

### Responsive
- **Desktop (>1024px):** Sidebar + main panel as designed
- **Tablet (768-1024px):** Sidebar collapses to hamburger. Comparison stacks vertically
- **Mobile (<768px):** Charts stack. Score card uses 2-column grid. Timeline gets horizontal scroll

### Accessibility
- Chart alt text on every Plotly figure
- ARIA landmarks: sidebar=navigation, main=main, score card=region
- Keyboard nav: arrow keys in trigger list, Enter to select
- Touch targets ≥44px
- WCAG AA color contrast (4.5:1 normal, 3:1 large)
- Color-blind safe: icons (✓/✗) alongside color coding

## Open Questions

- Which peril type to start with? (Drought is simplest — rainfall data is clean and abundant)
- Hosting: self-hosted demo, or deploy to a public URL from day one?
- Trigger schema: how much of Lloyd's trigger specification to formalize in v1?
- How to handle triggers where open data is sparse or low-resolution?
- License choice: MIT, Apache 2.0, or AGPL (to prevent proprietary forks)?

## Success Criteria

- A live dashboard where you can select a parametric trigger and see its Spearman basis risk score, historical back-test, and Lloyd's alignment rating
- At least 3 example triggers across different peril types populated in the registry
- A dev can understand the entire trigger audit in under 60 seconds
- Computation is reproducible: same trigger + same data = same scores, every time
